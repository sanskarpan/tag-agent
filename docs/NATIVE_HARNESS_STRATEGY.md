# TAG — Native Harness Strategy & Migration Plan (Go)

> **Updated strategic direction (supersedes `STACK_DECISION.md` for the long term).** Produced by a
> 13-agent competitor-grounded research workflow after leadership reframed Hermes as a *transitional*
> dependency and made **adoption / single-binary distribution the #1 priority**.

## TL;DR

**Recommendation:** Rewrite TAG as a native, single-static-binary, own-the-runtime agent harness in GO. Do NOT ship a standalone hardened-Python (PyApp/uv) launcher as a product — it is a distribution dead-end (artifact still carries the 54.7MB Hermes tarball + a bundled CPython, growing toward ~75-90MB) and risks the aider "bridge-becomes-destination" trap. Instead run a STRANGLER migration: the only "interim" work permitted is decoupling Hermes behind a wire/remote protocol, because that is simultaneously step 1 of the Go rewrite. Ship the native Go core (LLM + tool-exec + MCP client/server + session/agent-loop) first, keep the Python control plane running behind that wire protocol, and port the 103 commands incrementally. During transition keep a familiar entry point via an optional thin npm/pip wrapper that downloads the native binary (the Codex pattern), with curl|sh + Homebrew + winget/scoop as the primary channels and in-place self-update.

**Go vs Rust:** GO. On the top-weighted axis (adoption/distribution) Go and Rust are effectively tied for real users — once both are a single dependency-free file installed via curl|sh/brew and both cold-start in milliseconds, Rust's 2-5MB vs Go's 10-15MB size edge is below the threshold anyone notices or chooses on (adversarial adoption verdict: Rust 9.5 / Go 9.0). Go then wins the three remaining weighted criteria decisively for THIS team: (1) runtime-ownership feasibility (Go 9 / Rust 7) because Go uniquely has OFFICIAL GA SDKs for BOTH providers Hermes must serve — anthropic-sdk-go (v1-stable) and openai-go — plus the official, Google-co-maintained MCP go-sdk v1.0 (client AND server), whereas Rust's Anthropic SDK is fragmented with no canonical crate; (2) build velocity + maintainability against a documented bus-factor-of-one (Go 8.5 / Rust 6) — 5-15s builds and a days-not-months learning curve vs Rust's 1-3min builds and 3-6mo borrow-checker tax, which bites hardest in exactly the reach-parity phase; (3) channel breadth, where Go's crush ships the widest install matrix of any peer. Rust's genuine edges — smallest binary and mature native OS sandboxing (Landlock/seccomp/sandbox-exec) — are either invisible to users or deferrable: TAG already has container backends (docker/daytona/modal) in-repo for untrusted tool exec (PRD-028/005), so native syscall sandboxing is not a hard day-one runtime requirement. Reserve Rust ONLY if native OS-level sandboxing of untrusted code becomes a first-class, must-own requirement.

**Verdict:** Go native single binary via a strangler migration — 95% of Rust's distribution win at materially higher velocity and stronger official SDK coverage; skip the PyApp interim and instead decouple Hermes behind a wire protocol as step 1 of the Go rewrite.

**Runner-up:** RUST — the correct choice if, and only if, native OS-level sandboxing of untrusted tool/code execution (Landlock/seccomp/sandbox-exec) or no-GC streaming under heavy parallel `exec` fan-out becomes a hard, must-own runtime requirement. Its ecosystem for the runtime primitives is production-grade today (rmcp official MCP client+server ~4.7M downloads, ratatui, rusqlite/sqlx, tokio+axum, async-openai), it ships the smallest binary, and goose + Codex-rs are near-exact precedents. It loses here only on velocity-to-parity for a solo maintainer and the fragmented Anthropic SDK. Second runner-up is the hardened-Python interim, but strictly as time-boxed launcher plumbing, never as the end-state.

**Confidence:** High on the two headline calls: (1) native single-binary end-state over any Python packaging — near-certain given the unanimous 2026 competitor convergence, Claude Code deprecating its own npm channel, and Codex's 510x post-rewrite growth; (2) reject the standalone hardened-Python interim as a product — high confidence, its distribution cap is arithmetic (Hermes tarball + bundled CPython grows the artifact) and the strategic dead-end risk is well-evidenced by aider. Medium-high on GO over RUST specifically: Go wins 3 of 4 weighted criteria for this solo-maintainer, adoption-first context and has the stronger official SDK story, but this is a genuinely contestable call — the field is split (Codex/goose Rust vs crush Go) and the decision would flip to Rust if native OS-level sandboxing of untrusted execution is deemed a first-class must-own requirement, or if a second Rust-fluent maintainer materially lowers the velocity risk. Confidence would rise further after a one-week bubbletea streaming-TUI spike and a thin anthropic-sdk-go + MCP go-sdk agent-loop proof-of-concept, which are the two cheapest ways to validate the riskiest Go assumptions before committing the full port.

## Why native single-binary (the competitor evidence)

- Codex CLI is the strongest native precedent: OpenAI rewrote it from TypeScript/React/Node to Rust (codex-rs, ~95.6% Rust) within ~2 months of the April 2025 open-source launch. The four STATED reasons were (1) zero-dependency install because the Node v22+ requirement was 'frustrating or a blocker for some users', (2) lower memory / no runtime garbage collection so GC pauses never interrupt streaming mid-turn and startup is milliseconds (matters for `codex exec` in CI launching many agents), (3) native OS sandboxing (macOS sandbox-exec, Linux Landlock/seccomp) via already-shipping Rust bindings, and (4) a language-agnostic 'wire protocol' so TS/Python/other langs can extend the agent plus native MCP. Sources: infoq.com, github.com/openai/codex/discussions/1174.
- Codex adoption AFTER going native validates that distribution/UX drives growth: npm downloads went from 82,000 (Apr 2025) to 41.8M (May 2026) — ~510x; it hit 5M weekly active users and 90,000+ GitHub stars in under 14 months, with 840+ releases (nearly 2/day). Critically, OpenAI kept the TS version in parallel for bug fixes until Rust reached parity, and still ships an npm wrapper (@openai/codex) that downloads the native binary — the native binary is now the default. This shows a native rewrite need not abandon the familiar `npm i -g` entry point.
- block/goose is the purest single-binary model: written in Rust, ships as one self-contained native CLI binary (plus an Electron desktop app) for macOS/Linux/Windows. Install is `curl -fsSL .../download_cli.sh | bash`, Homebrew, or GitHub-release binaries (also deb/rpm/flatpak/win). Rust was chosen for speed, portability, and single-binary distribution with no central control plane/telemetry. ~45-50k GitHub stars, Apache-2.0, 15+ LLM providers, 70+ MCP extensions. In April 2026 Block DONATED goose to the Linux Foundation's new Agentic AI Foundation (repo moved block/goose -> aaif-goose/goose) — a neutral-governance move that signals an agent runtime maturing into shared infrastructure, exactly the 'own-the-runtime' end-state TAG aspires to.
- charmbracelet/crush is the Go counter-example and the most install-friendly of all peers: a single cross-platform binary <10MB (macOS/Linux/Windows/*BSD) with the WIDEST distribution matrix of any competitor — GitHub binaries, `go install`, Homebrew, npm (@charmland/crush), Arch (yay), Nix, Winget, Scoop. ~20.7k stars (Jan 2026). It integrates LSP for real code intelligence and MCP over http/stdio/sse, and inherits the archived Go 'opencode' lineage plus Charm's Bubble Tea TUI ecosystem. Go's small-static-binary + goroutine story delivers the same distribution win as Rust with materially higher build velocity — the key trade for a small team.
- cursor-cli / cursor-agent (Anysphere) is a closed-source NATIVE binary with zero Node/Python runtime requirement: install via `curl https://cursor.com/install -fsSL | bash`, Homebrew cask, or platform archives from downloads.cursor.com, with auto-update on by default. Sourcegraph's Amp distributes prebuilt native binaries from packages.ampcode.com behind both a `curl .../install.sh | bash` script AND an npm wrapper (@sourcegraph/amp). The consistent 2026 pattern across native tools: curl|sh + brew as the primary install, an optional npm/thin wrapper for discoverability, and self-contained binaries that need no interpreter.
- The counter-evidence matters: aider — the ONLY true language-peer to TAG (a Python CLI wrapping Python) — did NOT rewrite. It shipped a uv-based bootstrap (`aider-install` installs only uv, then `uv tool install` into an isolated env, auto-installing Python 3.12) to fix provisioning/dependency pollution in-ecosystem. Claude Code (TS/Node) and sst/opencode (TS on Bun) also stayed interpreted, betting on ecosystem reach. So native-binary is dominant among NEW/ambitious agent harnesses but not universal; the deciding variable is whether the tool must ship a bundled interpreter runtime. Every native peer (Codex, goose, crush, cursor) owns its runtime and ships NO bundled Python — which is precisely why their distribution win is clean.
- The repo already contains a prior stack analysis at /Users/sanskar/dev/test/tag/docs/STACK_DECISION.md that recommended staying Python. Its own reasoning explicitly names the escape hatch that leadership has now triggered: it says the verdict 'would flip toward a native (Go, not Rust) rewrite' IF 'Hermes were reimplemented or replaced by a non-Python runtime' or the center of gravity moved to long-lived high-concurrency servers. Since leadership has now declared Hermes a transitional dependency and TAG owning its own runtime as the north-star, the STACK_DECISION.md's own stated condition for choosing native is satisfied — and within native it leaned Go over Rust for a bus-factor-of-one team (velocity, PyO3 being irrelevant once Python coupling is gone).

## Rationale

The strategic reframe is decisive: with Hermes declared transitional and adoption the #1 goal, the escape hatch that TAG's own docs/STACK_DECISION.md named ("verdict flips to native, Go not Rust, if Hermes is replaced") is now triggered. Every 2026 exemplar devs praise (ripgrep, uv, gh, ollama, crush) and every ambitious own-the-runtime agent (Codex, goose, crush, cursor, Amp) converged on one small dep-free native binary — and Claude Code KILLED its own npm channel in Jan 2026 for exactly TAG's stated reasons. Codex's 510x download growth post-Rust-rewrite is direct proof that killing the interpreter dependency is an adoption unlock. TAG's current story is worst-in-class on the axis that matters most: a 52MB npm tarball gated on Node>=20 that shells out to a system Python 3.11-3.13 and builds a venv via live pip on first run (bin/tag.js:143-148) — three runtimes, a network-dependent first launch, two update paths, and Windows py-launcher shims. Between the two native options, the field is genuinely split (Rust: Codex/goose; Go: crush) so the language question is settled by team fit, not fashion. For a bus-factor-of-one maintainer hand-porting ~72 bespoke control-plane features (memory tiers, eval/judge, DAG queue, swarm, cron, OTel attribution) across a 103-command surface, velocity-to-parity IS the binding constraint, and Go wins it while still delivering the full distribution win. Go also uniquely has official GA SDKs for both Anthropic and OpenAI plus the official MCP SDK — de-risking runtime ownership to a port, not research. The interim is rejected as a product because it caps at ~40-60% of the total distribution win (Hermes tarball + bundled CPython actually grows the artifact) and its real hazard is strategic: removing acute install pain cheaply also removes urgency to fund the native rewrite. The one convergent interim action — Hermes-as-wire-service — is preserved because it is literally the first strangler step of the Go build, not throwaway.

## What it takes to OWN the runtime (Go subsystem map)

Owning the runtime is a de-risked 6-12 month port, not research — every load-bearing subsystem has a GA Go analog. Concretely: (1) LLM layer — replace openai==2.24.0/anthropic==0.86.0 with official anthropic-sdk-go (Messages.NewStreaming -> ssestream + Message.Accumulate(), tool use, prompt caching) and openai-go (Responses API, SSE, function calling, JSON-Schema structured outputs); reuse openai-go's base-URL override for OpenRouter/Ollama/OpenAI-compatible providers. (2) Agent loop — hand-roll native (call -> tool-dispatch -> observe -> repeat, ~200 LOC, the crush pattern); do NOT adopt eino/langchaingo as a core dependency. (3) MCP — official modelcontextprotocol/go-sdk v1.0 for both client (PRD-014 registry) and server (PRD-006 gateway); plan around known gaps (sampling-with-tools, experimental client OAuth). (4) Tool exec / sandbox — os/exec + goroutines for dispatch; lean on existing container backends (docker/daytona/modal, PRD-028/005) since Go lacks a mature Landlock/seccomp binding. (5) State — modernc.org/sqlite (pure-Go, CGO_ENABLED=0 keeps static cross-compile intact) ports the runs/steps/spans/memory/queue WAL+FTS5 schema; add sqlite-vec for semantic/vector memory. (6) Servers — net/http + goroutines replace fastapi/uvicorn for the webhook trigger (PRD-016/056) and gateway. (7) TUI — Charm bubbletea/lipgloss/bubbles replaces prompt_toolkit + the Hermes TUI (pressure-test the highest-throughput streaming view early per the opencode Bubble-Tea ceiling; keep a Zig/ratatui-backed renderer as fallback). (8) Observability — native go.opentelemetry.io/otel for GenAI-semconv export + cost attribution (PRD-012/013/041/048). (9) Release — goreleaser + cosign/Sigstore + SLSA provenance + macOS notarization; self-update in place. MIGRATION DISCIPLINE (non-negotiable): strangler pattern — expose Hermes over a wire protocol FIRST, ship the native Go core behind it, dual-maintain the Python surface until parity (the Codex playbook: keep the old runtime alive until the native one reaches parity), port commands incrementally so there is never a feature-frozen big-bang.

## Key risks

- Full rewrite of ~72 bespoke control-plane features (~32.7k Py LOC, 700+ tests) against a bus-factor-of-one maintainer is multi-quarter regardless of language — the dominant project risk is maintainer bandwidth. Mitigation: strict strangler, port command-by-command, dual-maintain until parity; consider adding a second maintainer.
- Go TUI performance ceiling: opencode ABANDONED Go+Bubble Tea over 'severe performance issues at scale' for AI-driven interactive UIs. Mitigation: prototype live token-streaming into a multi-pane dashboard on bubbletea in week one; if it hits the ceiling, back the renderer with Zig/ratatui behind the Go front-end.
- ML/embedding/vector-graph memory is Python's remaining moat (semantic memory w/ decay PRD-025, vector retrieval PRD-043, episodic/graph + community detection PRD-070). Go has no numpy/sentence-transformers-class local stack. Mitigation: degrade to provider embedding APIs + sqlite-vec; hand-roll graph algos on gonum — adds a network dependency and real porting cost.
- Native OS sandboxing for untrusted tool exec (PRD-028) is Go's structural soft spot (no Landlock/seccomp binding as mature as Rust's — the exact reason Codex/goose chose Rust). Mitigation: container/gVisor isolation via existing backends; if in-process syscall sandboxing becomes a hard requirement, this is the single trigger to reconsider Rust.
- Interim-bridge moral hazard: shipping a hardened-Python launcher removes acute install pain cheaply and thereby removes urgency to fund the native rewrite (aider is the literal cautionary tale). Mitigation: do NOT ship a standalone PyApp product; permit only the convergent Hermes-as-wire-service slice, time-boxed with an explicit kill date tied to a funded native milestone.
- Strangler scope-creep splitting the single SQLite DB across Python and Go reintroduces the read-modify-write race family the repo has spent bug-bashes hardening. Mitigation: keep exactly one writer/owner of the SQLite state during transition; the wire protocol, not shared DB files, is the seam.
- Provider-SDK/MCP spec churn (MCP 2026-07-28 deprecated roots/sampling/logging on a 12-month window; sampling-with-tools unimplemented). Mitigation: wrap SDKs behind a thin internal interface so spec moves are localized.

# Migration plan

**Approach:** Strangler-fig migration to a native single-static-binary Go harness that OWNS its runtime, sequenced so the distribution/UX win ships FIRST (before the runtime is fully ported) and the product never stops shipping. Two hard rules govern the whole migration: (1) there is exactly ONE writer/owner of the SQLite state at all times (the seam between Python and Go is a wire protocol, NEVER shared DB file access) — this protects the read-modify-write race hardening the repo earned across three bug bashes (see src/tag/core/db.py migrations); (2) every Hermes call goes through the three chokepoints already present in src/tag/core/profile.py (run_hermes L199, run_profile_hermes L210, run_profile_python L228) — collapsing all Hermes coupling behind one Go-facing wire interface is simultaneously the first strangler cut AND the first step of owning the runtime, so the interim work is convergent, not throwaway. The Go binary starts life as a thin front-door/launcher wrapping today's Python+Hermes over that wire protocol (instant distribution win), then absorbs subsystems inward — LLM layer, agent loop, MCP, tool exec, SQLite state, servers, TUI — porting the ~97-command surface (docs/FEATURES.md) command-by-command behind a compatibility harness, until Hermes and finally the Python control plane are deleted. Language is Go per the decision record: 95% of Rust's distribution win at materially higher velocity for a bus-factor-of-one maintainer, plus official GA anthropic-sdk-go + openai-go + Google-co-maintained MCP go-sdk v1.0 de-risk runtime ownership to a port rather than research.

**End state:** TAG is a single statically-linked Go binary (~10-15MB, CGO_ENABLED=0, ldflags -s -w), zero runtime dependencies, installed via `curl|sh` + Homebrew + winget/scoop with an optional thin npm/pip wrapper that merely downloads the native binary (Codex pattern) for discoverability. It owns its entire runtime: LLM calls (anthropic-sdk-go + openai-go, OpenAI-compatible base-URL for OpenRouter/Ollama), a hand-rolled agent loop (call -> tool-dispatch -> observe -> repeat, ~200 LOC, the crush pattern), MCP client AND server (modelcontextprotocol/go-sdk v1.0), tool execution (os/exec + goroutines, untrusted code isolated via the existing docker/daytona/modal container backends), session/run/step/span/memory/queue state (modernc.org/sqlite pure-Go with FTS5 + sqlite-vec for semantic/vector memory), long-lived servers (net/http + goroutines for webhook trigger + gateway + dashboard), a Charm bubbletea/lipgloss/bubbles TUI, and native OpenTelemetry GenAI-semconv export + cost attribution. The 52MB vendored Hermes tarball (src/tag/vendor/hermes-agent-upstream.tar.gz), bin/tag.js, the first-run venv/pip provisioning, and the entire Python control plane are gone. Cold start is milliseconds; install is one command with no Node, no system Python, no network-dependent first launch. Releases are goreleaser-driven with Sigstore/cosign keyless signing, SLSA provenance, macOS notarization, SBOM, and in-place self-update.

**Total effort:** Roughly 8-12 months of focused work for a bus-factor-of-one maintainer (aligns with the decision record's 6-12 month runtime-ownership estimate, with the upper bound reflecting the full ~97-command port plus TUI/memory risk). Phase-by-phase: Phase 0 ~3-5 wks; Phase 1 ~4-6 wks (distribution win lands here, ~2 months in); Phase 2 ~6-9 wks; Phase 3 ~8-12 wks; Phase 4 ~8-12 wks; Phase 5 ~3-4 wks. The dominant risk is maintainer bandwidth, not language or ecosystem — every load-bearing subsystem has a GA Go analog, so this is a de-risked port, not research. Adding a second maintainer would materially compress Phases 3-4 (the reach-parity long tail) where the work parallelizes cleanly across independent command modules. Critically, the top-weighted adoption win is realized at the END OF PHASE 1 (~2 months), long before full parity, because the native binary fronts the transitional Python backend — so business value is front-loaded and the remaining ~6-10 months hollow out the runtime behind an already-shipping native install experience.

## Guiding principles

- Keep shipping, always. Every phase ends with a release users can install; there is never a feature-frozen big-bang window. Dual-maintain the Python surface until each command reaches Go parity (the Codex playbook: TS kept alive until Rust hit parity).
- Win distribution FIRST, port runtime SECOND. The native binary must be the install artifact before the runtime is fully ported — the adoption unlock (Codex saw 510x download growth post-native) comes from killing the interpreter dependency, and it can front a Python backend transitionally.
- One SQLite writer at all times. The Python<->Go seam is a wire protocol (local HTTP/JSON-RPC over loopback or stdio), never a shared DB file. Splitting the single DB across two languages would reintroduce the read-modify-write race family the repo hardened across three bug bashes.
- Funnel Hermes through the existing chokepoints. All coupling already flows through run_hermes/run_profile_hermes/run_profile_python in src/tag/core/profile.py — collapse those behind ONE wire interface so replacement is localized and measurable.
- Wrap external SDKs behind a thin internal provider/MCP interface so 2026 spec churn (MCP 2026-07-28 deprecations, sampling-with-tools gaps) stays localized to one adapter.
- Parity is proven by a golden-transcript harness, not by trust. Every command has recorded stdin/args/env -> stdout/exit/DB-delta fixtures captured from the Python impl; the Go impl must replay them byte-for-contract before the Python version is deleted.
- Prefer hand-rolled over frameworks for the agent loop (crush/goose precedent). Do NOT adopt eino/langchaingo as a core dependency — TAG's 72 shipped features are bespoke control-plane logic that must be hand-ported in any language regardless.
- Time-box and kill-date every interim scaffold. The Python backend behind the wire protocol has an explicit retirement milestone tied to funded parity, so the bridge never becomes the destination (aider is the cautionary tale).

## First slice

Phase 0's convergent cut: refactor the three Hermes chokepoints in src/tag/core/profile.py (run_hermes L199, run_profile_hermes L210, run_profile_python L228) to route through a single versioned RuntimeService wire interface, backed by a thin Python 'hermes-runtimed' adapter, and build the golden-transcript capture harness across the ~97 commands. This is the smallest change that (a) eliminates inline-Python-string exec into the Hermes venv, (b) establishes the one-writer SQLite contract, and (c) creates the exact seam a Go core will later speak — so it is step one of BOTH the strangler cut and owning the runtime, throwaway-free. In parallel, run the week-one de-risking spike: a bubbletea live-token streaming multi-pane prototype plus a ~50-line anthropic-sdk-go + MCP go-sdk agent-loop proof-of-concept, to validate the two riskiest Go assumptions before committing the full port.

## Phases

### Phase 0 — Decouple Hermes behind a wire protocol (the convergent first cut)  (3-5 weeks)
**Goal:** Collapse ALL Hermes coupling behind a single, versioned wire interface so Python no longer inlines Python strings into the Hermes venv, and so a future Go core can speak the same protocol. This is simultaneously the last necessary Python-side refactor and the first step of owning the runtime.

**Scope:**
- Define an internal RuntimeService interface (JSON-RPC/HTTP over loopback or stdio) covering the operations the three chokepoints perform: run agent turn, stream tokens, dispatch tool, auth/credential ops.
- Refactor src/tag/core/profile.py run_hermes (L199), run_profile_hermes (L210), run_profile_python (L228) to route through a single HermesRuntimeClient implementing RuntimeService — eliminate inline `python -c` string exec into the Hermes venv.
- Stand up a thin Python 'hermes-runtimed' adapter process that exposes Hermes over the wire protocol; the control plane becomes its sole client.
- Establish the SQLite single-writer contract: document that only the control plane process touches the DB; the runtime service returns data over the wire, never opens the DB file.
- Build the golden-transcript capture harness: instrument the ~97 commands (docs/FEATURES.md) to record args/env/stdin -> stdout/exit-code/DB-delta fixtures in an isolated TAG_HOME sandbox (per the v0.8.1 runtime-audit memo: verify by RUNNING, not just pytest).

**Keeps shipping:** Yes — behavior is identical to users; this is an internal refactor released as a normal patch version. No install-story change yet.
**Risks:** Hidden stateful coupling in inline Python strings that assumes shared process memory with Hermes.; Streaming semantics (token-by-token) must survive the wire boundary without added latency.
**Exit:** All Hermes invocations flow through RuntimeService; zero inline-Python-string exec into the Hermes venv remains; golden transcripts captured for 100% of commands and passing against current Python impl; existing 678-test suite green.

### Phase 1 — Native Go launcher/front-door (win distribution UX BEFORE the runtime is ported)  (4-6 weeks)
**Goal:** Ship the native single binary as the install artifact NOW. The Go binary is initially a thin front-door: it parses args, owns the TUI shell and self-update, and forwards command execution to the still-Python control plane + Hermes-runtimed over the wire protocol from Phase 0. This delivers the #1 adoption lever (single dep-free binary, curl|sh/brew) months before the runtime port completes.

**Scope:**
- Scaffold the Go binary (cobra/urfave CLI), embedding the RuntimeService wire client from Phase 0.
- goreleaser pipeline: cross-compile macOS(arm+intel)/Linux(musl+glibc)/Windows via CGO_ENABLED=0 + GOOS/GOARCH; produce checksums, SBOM, Sigstore/cosign signing, SLSA provenance, macOS notarization.
- Distribution channels: curl|sh installer + Homebrew tap/cask + winget/scoop; optional thin npm wrapper (@tag/cli style) and pip shim that download the native binary (Codex/@openai/codex pattern) to preserve the familiar entry point during transition.
- In-place self-update (`tag self update`), background update check.
- TRANSITIONAL packaging bridge: the Go binary bootstraps the Python backend it needs. To avoid re-shipping the Node+first-run-pip mess, the backend (control plane + Hermes) is delivered as a PyApp/uv relocatable-Python sidecar fetched/managed by the Go binary — NOT via bin/tag.js. This kills Node>=20 and the live-pip first-run immediately; the 52MB Hermes tarball still ships inside the sidecar (honest size cap ~55-90MB until the runtime is ported inward).
- Delete bin/tag.js and the package.json engines>=20 dependency from the primary install path.

**Keeps shipping:** Yes — this IS a headline release (v1.0 native installer). Users get the distribution win now even though Go owns no runtime yet.
**Risks:** Bubbletea streaming-TUI ceiling (opencode abandoned Go+Bubble Tea at scale) — prototype the highest-throughput live-token multi-pane view in week one; fallback is a Zig/ratatui-backed renderer behind the Go front-end.; Sidecar Python bootstrap must be robust offline and behind proxies; test PyApp relocatable-CPython packaging on all three OSes early.; Artifact size temporarily grows (bundled CPython + Hermes tarball) — message this as transitional and track it down each subsequent phase.
**Exit:** `curl|sh` / `brew install tag` yields a working native binary with zero Node and zero system-Python requirement and no network-dependent first-run for the control plane; self-update works; all ~97 commands run through the Go front-door with golden transcripts green; signed/notarized artifacts published.

### Phase 2 — Own the LLM layer + agent loop natively in Go  (6-9 weeks)
**Goal:** Move the core agent runtime (LLM calls, streaming, tool dispatch, agent loop) from Hermes into native Go, so the Go binary stops needing Hermes for the primary agent execution path.

**Scope:**
- Implement the provider layer behind a thin internal interface: anthropic-sdk-go (Messages.NewStreaming -> ssestream + Message.Accumulate(), tool use, prompt caching) and openai-go (Responses API, SSE, function calling, JSON-Schema structured outputs); reuse openai-go base-URL override for OpenRouter/Ollama/OpenAI-compatible providers (mirrors TAG's 18-source credential import).
- Hand-roll the agent loop natively (call -> tool-dispatch -> observe -> repeat, ~200 LOC, crush pattern) — no eino/langchaingo core dependency.
- Port credential/auth handling off hermes_cli.auth to native Go.
- Route the main run/chat/agent commands to the native loop; leave niche/less-used agent behaviors on Hermes-runtimed via the wire protocol (strangler — port command-by-command).
- Extend golden-transcript harness to cover streaming output ordering and tool-call round-trips.

**Keeps shipping:** Yes — released incrementally; ported commands flip to native behind the same CLI, un-ported ones keep working via Hermes wire service.
**Risks:** Reproducing exact Hermes prompt/format/tool-schema behaviors — golden transcripts are the guardrail.; Prompt-caching and provider-specific streaming quirks differ between SDKs; localize in the adapter.

## Distribution-win plan

Ship the native Go single binary as the install artifact in Phase 1, BEFORE the runtime is ported, fronting the transitional Python/Hermes backend over the Phase-0 wire protocol. Primary channels: `curl -fsSL https://.../install.sh | sh` + Homebrew tap/cask + winget/scoop, mirroring crush's widest-in-class matrix. Keep a familiar entry point via an OPTIONAL thin npm wrapper (downloads the native binary, the @openai/codex pattern) and pip shim — discoverability only, never a runtime dependency. Build: goreleaser + CGO_ENABLED=0 + modernc.org/sqlite for trivial cross-compile of macOS(arm+intel)/Linux(musl+glibc)/Windows from one machine; ldflags -s -w. Size trajectory: transitional artifact is ~55-90MB in Phase 1 (bundled relocatable CPython + 52MB Hermes tarball inside the sidecar — honestly larger before it gets smaller), shrinking each phase as subsystems move into Go, reaching the true ~10-15MB single binary in Phase 5. Immediate Phase-1 wins even at large size: kills Node>=20, kills system-Python 3.11-3.13 detection + Windows py-launcher probing, kills the network-dependent first-run pip build, collapses two update paths to one. Auto-update: in-place `tag self update` + background check. Supply chain: Sigstore/cosign keyless signing + SLSA provenance + macOS notarization + SBOM — one signed artifact with a far smaller blast radius than today's npm-tarball-plus-live-pip model (relevant given pyproject.toml already exact-pins deps post Shai-Hulud). The bin/tag.js Node launcher and package.json engines>=20 are removed from the primary path in Phase 1.

## How Hermes gets replaced

Incrementally, via the strangler seam, never a big-bang. (1) Phase 0 funnels 100% of Hermes coupling through a single RuntimeService wire interface at the three profile.py chokepoints, replacing inline `python -c` exec into the Hermes venv with a versioned protocol and standing up a thin Python hermes-runtimed adapter as the transitional backend. (2) Phase 2 implements the core runtime natively in Go — provider layer (anthropic-sdk-go + openai-go behind a thin internal interface) and a hand-rolled ~200-LOC agent loop — and flips the primary agent commands (run/chat/agent/autonomous) to native, leaving niche behaviors on hermes-runtimed. (3) Phase 3 ports MCP (go-sdk v1.0 client+server), tool exec (os/exec + goroutines, untrusted code to existing docker/daytona/modal backends), and the SQLite state (modernc.org/sqlite + FTS5 + sqlite-vec), executing a one-time DB-ownership handoff so Go is the sole writer. (4) Phase 4 ports the servers (net/http replacing fastapi/uvicorn), native OTel, and the remaining bespoke commands. (5) Phase 5 deletes the Hermes tarball, the hermes-runtimed adapter, and the entire Python backend once production-soaked parity is proven. Throughout: every command carries golden-transcript fixtures (args/env/stdin -> stdout/exit/DB-delta captured from the Python impl) that the Go impl must replay before the Python version is deleted; un-ported commands keep running via the wire service so the product never feature-freezes. Parity vs the ~97-command surface is proven by the golden-transcript suite PLUS isolated-TAG_HOME sandbox runs (per the v0.8.1 audit memo, because the unit suite masks dispatch-layer bugs) — not by the unit suite alone.

## Explicit non-goals

- Do NOT ship a standalone hardened-Python (PyApp/uv) launcher as the PRODUCT end-state — it caps at ~40-60% of the distribution win (bundled CPython + Hermes tarball actually grows the artifact) and risks the aider bridge-becomes-destination trap. PyApp appears ONLY as an internal, time-boxed, kill-dated backend-sidecar mechanism inside the Go binary during Phases 1-4, never as a user-facing deliverable.
- Do NOT split the single SQLite database across Python and Go via shared file access — the seam is the wire protocol only; there is exactly one writer at all times, or the read-modify-write race family the repo hardened across three bug bashes returns.
- Do NOT adopt eino/langchaingo (or any agent framework) as a core dependency — hand-roll the ~200-LOC agent loop (crush/goose precedent); TAG's 72 features are bespoke control-plane logic that must be hand-ported regardless.
- Do NOT choose mattn/go-sqlite3 (CGO) — it re-breaks CGO_ENABLED=0 static cross-compile; use pure-Go modernc.org/sqlite even at ~2x query cost.
- Do NOT attempt native OS-level syscall sandboxing (Landlock/seccomp) in Go for untrusted tool exec — Go's binding is immature (the reason Codex/goose chose Rust); rely on the existing docker/daytona/modal container backends. If in-process syscall sandboxing later becomes a first-class must-own requirement, that single trigger is the reason to reconsider Rust.
- Do NOT feature-freeze for a big-bang rewrite — port command-by-command behind golden transcripts, dual-maintaining the Python surface until each command reaches parity.
- Do NOT trust the unit suite as the parity gate — verify features by RUNNING commands in isolated TAG_HOME sandboxes (per the v0.8.1 runtime-audit memo, which found ~30 dispatch-layer bugs masked by green unit tests).
- Do NOT reconsider Rust unless native OS-level sandboxing of untrusted execution or no-GC streaming under heavy parallel exec fan-out becomes a hard must-own requirement — those are the only documented flip conditions.
- Do NOT delete Hermes or the Python control plane until a production stabilization soak (not just green CI) confirms parity.
