# TAG — Python→Go Migration Research

> Primary research pass grounding the Go migration. 13 agents cloned + audited real source
> (sst/opencode, charmbracelet/crush, Hermes v0.16/latest) and web-researched the 2026 market,
> the Go library stack per subsystem, and concrete solutions to every migration risk.

## Executive summary

The Go migration is well-founded and the ecosystem is ready. All three serious native agent CLIs (Codex→Rust, goose→Rust, crush→Go) abandoned dynamic-runtime languages for exactly TAG's stated reasons (install friction, GC pauses in long-running loops), and charmbracelet/crush is a near-drop-in Go template: bubbletea v2 TUI, official modelcontextprotocol/go-sdk, pure-Go SQLite, a client/server daemon split, and a one-file GoReleaser pipeline. The definitive stack is: hand-rolled agent loop over the two official GA SDKs (anthropics/anthropic-sdk-go v1.55, openai/openai-go v3.41) behind a thin in-house provider interface; modelcontextprotocol/go-sdk v1.6.1 (GA, client+server) for MCP; modernc.org/sqlite (pure-Go, FTS5 compiled in) as the single state store; gonum for the entity graph; Charm v2 (bubbletea/lipgloss/bubbles/glamour) for the TUI; a Landlock+seccomp→Docker→gVisor→Firecracker isolation ladder for sandboxing; net/http (1.22 ServeMux)+chi+go-sse for servers; koanf v2 for config; go.opentelemetry.io/otel for tracing; and GoReleaser+cosign+SLSA for distribution. Three prior "hard" risks are now stale and largely retired: the Bubble Tea perf ceiling (fixed by the v2 Cursed Renderer, proven in crush), Go's lack of a mature Landlock/seccomp binding (go-landlock + elastic/go-seccomp-bpf are viable in 2026), and MCP OAuth/sampling-with-tools gaps (closed in go-sdk v1.4–v1.6). The single genuine capability loss is local offline embeddings (no numpy/torch peer), which is bounded because TAG's shipped memory code already degrades to FTS5. The dominant residual risk is non-technical: solo-maintainer bandwidth against ~72 features / ~32.7k Python LOC, mitigated by a strict strangler-fig migration behind a golden-transcript harness with a single atomic SQLite-ownership handoff. Adopt opencode's architectural boundaries (typed server-is-the-product API, provider-as-data+protocol, context-epoch cache model) and Hermes's declarative ProviderProfile + evidence-based verification — but port none of their runtime implementations (Effect-TS / Python god-files).

## Biggest surprises (things that change the plan)

- The famous opencode Go TUI is GONE — the entire opencode stack is now TypeScript/SolidJS (@opentui) over a Zig core, with zero .go files in packages/tui. The 'opencode abandoned Go for performance' story is therefore NOT evidence that native/Rust beats Go for TUIs; it is evidence the pre-v2 Bubble Tea renderer was slow, a ceiling that Bubble Tea v2 has since retired and crush proves in production.
- Three of the strategy doc's headline 'hard' risks are already stale in TAG's favor: (a) Go's 'no mature Landlock/seccomp binding' — go-landlock + elastic/go-seccomp-bpf are viable in 2026; (b) MCP client-side OAuth and sampling-with-tools 'gaps' — both shipped in the official go-sdk v1.4–v1.6; (c) the Bubble Tea perf ceiling — fixed by the v2 Cursed Renderer.
- TAG's own memory 'moat' is far narrower than its PRDs imply. The SHIPPED entity_graph.py is only union-find/connected-components (a ~30-line Go port), NOT the Louvain/PPR/LLM pipeline PRD-070 describes; and semantic search is brute-force cosine over a JSON column plus hand-rolled BM25/RRF — all plain arithmetic that ports 1:1 with no numpy/torch. Port the shipped behavior for parity, treat gonum Louvain/PPR as an opt-in upgrade.
- modernc.org/sqlite compiles FTS5 (plus RTree/JSON1/Session/GeoPoly) in BY DEFAULT, so the entire PRD-025/066 keyword+hybrid-search path is a zero-extra-work, CGO-free port — one of the biggest de-risking findings for the single-static-binary goal.
- charmbracelet/crush is not just a reference but a near-complete Go blueprint — client/server daemon, unified builtin+MCP tool interface, permission gate, official MCP SDK, pure-Go SQLite, and a one-file GoReleaser fan-out to 6 package managers — while shipping a <10MB static binary (vs opencode's Bun-embedded 50-110MB). Its FSL-1.1-MIT license, however, blocks direct code vendoring.
- Hermes's real agent loop is fully SYNCHRONOUS and hand-rolled (conversation_loop.py), which directly validates the ~200-LOC Go loop plan as faithful rather than a simplification — the intimidating complexity lives in god-files (gateway/run.py ~19.7k LOC) TAG explicitly should NOT port.
- The strongest queue/DAG libraries in Go (River=Postgres, asynq=Redis) both BREAK the single-binary mandate — there is no drop-in pure-Go embedded DAG-capable durable queue in 2026, so PRD-008/033/105 require building a bespoke SQLite-backed goroutine scheduler. This is one of the few places that is genuine engineering, not a library drop-in.
- Codex's own proposal to slim its npm package from ~100MB to ~20MB (issue #2766) was closed as NOT planned — so TAG should follow the esbuild/optionalDependencies pattern, not Codex's actual (bloated) npm packaging.
- Hermes 0.18's evidence-based verification ('done means proven by running the project's real checks, not claimed') and Mixture-of-Agents-as-a-virtual-provider are novel, portable differentiators that neither opencode nor the Go peers have — potential leapfrog features rather than parity work.

## Risk solutions (all 7 — 3 were stale and are retired)

### (1) Go TUI performance ceiling — opencode abandoned Go+Bubble Tea over 'severe performance issues at scale' for AI-driven interactive UIs (strategy doc Key Risks; Phase 1/4 residual risk).
**Solution:** Adopt the Charm v2 stack (charmbracelet/bubbletea v2 + lipgloss v2 + bubbles v2 + glamour). Feed streamed tokens into a bubbles `viewport` driven by the Elm Update loop (Msg-per-delta), never full-frame reprints; render markdown incrementally via glamour. De-risk in week one exactly as the plan states: build a live multi-pane token-streaming prototype on bubbletea v2 and confirm it holds at full throughput before committing Phase 4 TUI work. Keep an OpenTUI-style Zig/ratatui-backed renderer behind the Go front-end only as a documented fallback that the spike would trigger — do NOT build it speculatively.

**Why it works:** The opencode abandonment predates and does not generalize: opencode dropped Go+Bubble Tea but moved to a TypeScript/SolidJS front-end over a Zig core (OpenTUI) — it is NOT evidence that Rust/native beats Go for TUIs, it is evidence that the *pre-v2* Bubble Tea renderer was slow. That specific ceiling has been retired: Bubble Tea v2 (GA, v2.0.0) ships the 'Cursed Renderer,' rebuilt on ncurses diff algorithms, reported as orders-of-magnitude fewer bytes written / lower bandwidth, and adds Mode 2026 synchronized output to kill tearing. Decisively, the direct Go peer — Charm's own Crush, an AI coding agent with exactly TAG's long-running high-churn streaming profile — ran v2 in production for months under SSH latency before GA. A same-class Go agent TUI shipping at scale is the strongest possible counter to the opencode anecdote.
**Tools:** charmbracelet/bubbletea v2 (MIT, GA v2.0.0, 2026 — Cursed Renderer), charmbracelet/lipgloss v2 (MIT), charmbracelet/bubbles v2 (MIT — viewport/spinner/progress/table), charmbracelet/glamour (MIT — incremental markdown), reference: charmbracelet/crush (Go AI agent proving the pattern), fallback only: OpenTUI (Zig core) / ratatui behind the Go front-end
**Residual:** Pin ambiguity/breaking API churn v1->v2 (Cmd/Msg, renderer, program options) — confirm the exact pinned tag at build time. Absolute worst case (spike fails at TAG's specific multi-pane throughput) still costs a renderer swap, but crush's production use makes that low-probability.
**Confidence:** High — the stated ceiling is a stale, pre-v2, non-Go-generalizable anecdote; the fix is a GA, permissively-licensed, production-proven Go stack. Confidence becomes near-certain after the week-one spike.

### (2) ML/embedding/vector-graph memory is Python's remaining moat — Go has no numpy/torch/sentence-transformers-class local stack (semantic memory w/ decay PRD-025, vector retrieval PRD-043/072, entity graph + community detection PRD-070).
**Solution:** Split the subsystem: (a) STORE/SEARCH/GRAPH port mechanically to GA Go — modernc.org/sqlite (pure-Go, FTS5 compiled in) for semantic_memories + FTS5 MATCH; confidence-decay is pure arithmetic; hybrid search (BM25 k1=1.5/b=0.75 + RRF k=60) is already hand-rolled arithmetic in the Python code; brute-force cosine over embeddings stored in a SQLite BLOB/JSON column is a ~20-line loop at TAG's hundreds-to-low-thousands-per-profile scale (no ANN engine needed); entity graph ports the SHIPPED union-find/connected-components (trivial), with gonum.org/v1/gonum providing the PRD-070 aspirational upgrades (community.Modularize = Louvain, network.PageRank) plus a ~40-line hand-rolled personalized-PageRank power iteration. (b) EMBEDDINGS (the only real gap): default to a provider embedding API behind a thin interface (openai-go / Voyage / Cohere / Gemini) and keep the EXISTING FTS5 graceful-degradation as the offline path; offer pure-Go local embeddings (nlpodyssey/cybertron for MiniLM) behind a build tag for air-gapped users who accept the perf hit.

**Why it works:** The moat is far narrower than the PRD prose implies: TAG's SHIPPED implementation is already brute-force cosine over a JSON column + hand-rolled BM25/RRF + union-find — none of which needs numpy/torch; they are plain arithmetic and SQL that port 1:1 to Go. FTS5 is compiled into modernc.org/sqlite by default, so keyword search and the whole PRD-066 candidate-retrieval path are zero-effort. The ONLY genuine loss is local offline sentence-transformers embeddings — and the Python code already degrades to FTS5 when embeddings are unavailable, so that same fallback simply becomes the primary offline path. gonum closes PRD-070's aspirational (not-yet-shipped) Louvain/PageRank gap natively.
**Tools:** modernc.org/sqlite (BSD-3, GA, pure-Go, FTS5+RTree+JSON1 built in, CGO_ENABLED=0), gonum.org/v1/gonum graph/community.Modularize + graph/network.PageRank (BSD-3, GA), openai-go embeddings / Voyage / Cohere / Gemini HTTP (provider embeddings), nlpodyssey/cybertron (BSD-2, pure-Go MiniLM, build-tag/offline option), optional scale-up: ncruces/go-sqlite3 + sqlite-vec (CGO-free WASM) or philippgille/chromem-go — only past ~100k vectors
**Residual:** Embedding path adds a network dependency + per-embed cost + an offline failure mode on mem add/search; dimension migration (384-dim MiniLM -> 1536-dim provider) needs a rebuild_embeddings migration + a model/dim tag in vector_store_meta and a mismatched-dim guard. gonum lacks Leiden (already a PRD non-goal) and built-in personalized PageRank (hand-rolled), and has a known Louvain infinite-loop on tiny deltaQ — cap iterations. cybertron is low-maintenance and slower than PyTorch.
**Confidence:** Medium-high — store/search/graph are low-risk mechanical ports onto GA libraries; risk is bounded to embeddings, which already has a shipped graceful fallback and stays viable at TAG's scale without any ANN engine.

### (3) Native OS sandboxing for untrusted tool exec (PRD-028) is Go's structural soft spot — 'no Landlock/seccomp binding as mature as Rust's,' the exact reason Codex/goose chose Rust; the single documented trigger to reconsider Rust.
**Solution:** Build PRD-028's isolation LADDER with native Go/host primitives, degrading gracefully by capability: (1) restricted tier becomes a REAL in-process boundary via landlock-lsm/go-landlock (filesystem) + elastic/go-seccomp-bpf (syscall filter, CGO-free) + google/nftables (egress, PRD-094) — replacing PRD-028's admittedly-fake fnmatch backend; (2) container tier via the official docker/docker moby client; (3) stronger tier via gVisor/runsc as a Docker runtime; (4) strongest/GPU tier via firecracker-go-sdk microVMs. TAG orchestrates these; it does not embed them. Continue leaning on the existing docker/daytona/modal backends already in-repo for untrusted exec, exactly as the strategy doc allows.

**Why it works:** The premise is now stale. The strategy doc's 'Go has no mature Landlock/seccomp binding' was true when Codex/goose chose Rust, but landlock-lsm/go-landlock is actively maintained with a June-2026 release and multi-ABI support, and elastic/go-seccomp-bpf is GA, CGO-free, and battle-tested in Elastic Beats. Landlock (kernel-object access control) + seccomp-bpf (syscall filtering) are complementary and together give a container-grade in-process boundary with no root/namespace/cgroup — a strict UPGRADE over PRD-028's fnmatch 'not a security boundary.' gVisor is itself written in Go. Crucially, strong isolation (Docker/gVisor/Firecracker) is a HOST dependency in ANY language — Rust cannot embed it either — so this is not a Go-specific deficit, and TAG already ships container backends, so day-one strong isolation is not a runtime-ownership blocker.
**Tools:** landlock-lsm/go-landlock (MIT, Linux 5.13+, actively maintained 2026), elastic/go-seccomp-bpf (Apache-2.0, GA, CGO-free), google/nftables (Apache-2.0, pure-Go netlink — PRD-094 egress), docker/docker moby client (Apache-2.0), google/gvisor runsc (Apache-2.0, Go-native runtime), firecracker-microvm/firecracker-go-sdk (Apache-2.0 — GPU/VFIO, PRD-093), existing in-repo docker/daytona/modal backends
**Residual:** Landlock/seccomp/nftables/gVisor/Firecracker are all Linux-only (Firecracker needs /dev/kvm); on macOS/Windows the only native tiers are plain-subprocess or Docker Desktop, so the ladder must feature-detect and degrade cleanly. Landlock network confinement is still incomplete (ABI ~v9) — use nftables/netns for egress, not Landlock. If in-process syscall sandboxing of untrusted code later becomes a first-class must-own requirement across all OSes, that remains the documented Rust flip trigger.
**Confidence:** Medium-high — the 'no mature binding' claim is now outdated (go-landlock + go-seccomp-bpf are viable in 2026), and strong isolation is a host dependency independent of language. Residual is the Linux-only reality of kernel sandboxing, which the ladder handles by design.

### (4) Maintainer bandwidth / bus-factor-of-one — full rewrite of ~72 bespoke control-plane features (~32.7k Py LOC, 700+ tests) across ~97-103 commands is multi-quarter; the dominant project risk.
**Solution:** Enforce the strategy doc's non-negotiable discipline: strict strangler-fig migration, porting command-by-command behind a golden-transcript harness (recorded args/env/stdin -> stdout/exit/DB-delta fixtures captured from the Python impl that the Go impl must replay before the Python version is deleted), dual-maintaining the Python surface until each command hits parity. Front-load the adoption win at end of Phase 1 (~2 months) so business value lands long before full parity. Choose Go specifically to minimize the bandwidth tax: 5-15s builds and a days-not-months learning curve vs Rust's 1-3min builds and 3-6mo borrow-checker tax, plus official GA SDKs (anthropic-sdk-go, openai-go, MCP go-sdk v1.x) that reduce runtime ownership to a PORT not RESEARCH. Prefer hand-rolled over frameworks (no eino/langchaingo) so there is one obvious way to do things and low onboarding cost for a future second maintainer.

**Why it works:** Bandwidth risk is attacked on three axes: (a) velocity — Go's fast builds and low ceremony directly buy back maintainer time in exactly the reach-parity long tail where the risk bites hardest, and the port parallelizes cleanly across independent command modules if a second maintainer is added; (b) never-freeze — the strangler + dual-maintain means the product ships every phase and there is no big-bang window that could strand a solo maintainer; (c) correctness leverage — golden transcripts + isolated-TAG_HOME sandbox runs (per the v0.8.1 audit memo, which found ~30 dispatch bugs the unit suite masked) let one person verify parity mechanically instead of by hand. Codex is the precedent: OpenAI kept TS alive until Rust reached parity.
**Tools:** strangler-fig migration pattern (per NATIVE_HARNESS_STRATEGY.md phases), golden-transcript capture/replay harness (Phase 0 deliverable), isolated TAG_HOME sandbox verification (v0.8.1 audit discipline), Go toolchain (5-15s builds), cobra/urfave CLI, official GA SDKs to avoid research: anthropic-sdk-go, openai-go/v3, modelcontextprotocol/go-sdk, goreleaser (one-command cross-compile release)
**Residual:** Even at Go's velocity this is an 8-12 month solo effort; the long tail of niche command flags is easy to under-scope (the transcript harness is the completeness ledger, but only if fixtures are truly exhaustive). The single highest-leverage mitigation — adding a second maintainer — is outside engineering's control. Language choice reduces but cannot eliminate the bus-factor.
**Confidence:** Medium — the discipline is sound and Go materially lowers the per-command cost, but the raw scope against one person remains the dominant, partly non-technical risk.

### (5) Interim-bridge moral hazard — shipping a hardened-Python launcher removes acute install pain cheaply and thereby removes urgency to fund the native rewrite (aider is the literal cautionary tale).
**Solution:** Ship NO standalone PyApp/uv launcher as a user-facing product. Permit only the ONE convergent slice: decouple Hermes behind a versioned RuntimeService wire protocol (Phase 0), which is simultaneously step one of the Go rewrite, not throwaway. During Phases 1-4 the relocatable-CPython + Hermes backend exists ONLY as an internal, kill-dated sidecar fetched/managed by the Go binary — never a deliverable — and its retirement (Phase 5) is gated to a funded native-parity milestone plus a production soak, not just green CI. Realize the top-weighted distribution win at end of Phase 1 via the native Go front-door so the incentive to keep going is structural, not willpower.

**Why it works:** It removes the hazard's mechanism rather than relying on discipline: the interim never becomes a comfortable end-state because (a) there is no shippable Python-launcher product to get attached to, (b) the only interim work (Hermes-as-wire-service) is convergent — deleting it is a Phase-5 line item, not a rewrite, so it is throwaway-free, and (c) tying the kill-date to a funded milestone converts 'urgency' from a motivational variable into a governance gate. aider became a dead-end precisely because its uv bootstrap WAS the product; TAG structurally forbids that by making the native binary the install artifact from Phase 1.
**Tools:** Phase-0 RuntimeService wire interface (JSON-RPC/HTTP over loopback or stdio), thin Python 'hermes-runtimed' adapter (transitional backend, kill-dated), PyApp/uv relocatable-CPython — internal sidecar ONLY, never user-facing, Go front-door binary (cobra/urfave) shipping in Phase 1, explicit kill-date milestone tied to funded native parity + production soak
**Residual:** Governance risk, not technical: if the funded native milestone slips or funding evaporates, the 'transitional' sidecar can quietly ossify despite the kill-date. The mitigation is organizational commitment, which engineering cannot fully guarantee. Sidecar size temporarily grows the artifact (~55-90MB) before it shrinks — must be messaged honestly as transitional.
**Confidence:** High on the mechanism (no standalone launcher + convergent-only interim removes the hazard by construction); medium on the residual governance dependency.

### (6) Splitting the single SQLite DB across Python and Go during the strangler migration reintroduces the read-modify-write race family the repo hardened across three bug bashes.
**Solution:** Maintain exactly ONE writer/owner of the SQLite state at all times. The Python<->Go seam is the wire protocol (local HTTP/JSON-RPC over loopback or stdio) — NEVER shared DB file access. In Phases 0-2 the Python control plane is the sole writer and the runtime service returns data over the wire without ever opening the DB file. Phase 3 executes a single, atomic, one-time DB-ownership handoff to Go, after which Go is the sole writer and the single-writer contract lives in Go. Preserve the existing serialized-writer discipline (flock + os.Rename atomic RMW, per core/config.py) on whichever side owns the DB — the race class is language-independent.

**Why it works:** The RMW race is a concurrency property of multiple writers, not a language property, so the only durable fix is architectural: keep the number of writers at one and make the cross-language boundary a message-passing seam, not shared mutable file state. A wire protocol enforces that data crosses as request/response, structurally preventing two processes from doing interleaved read-modify-write on the same rows. The handoff being a single atomic milestone (not a gradual dual-write period) means there is never a window where both languages write — which is exactly when the race would recur. Go's gofrs/flock is also a cross-platform upgrade over Python's fcntl path (which no-ops on Windows), so the owned-side discipline actually strengthens post-migration.
**Tools:** Phase-0 RuntimeService wire protocol (the only Python<->Go seam), modernc.org/sqlite (pure-Go, WAL, single-writer semantics identical to Python sqlite3), gofrs/flock + os.Rename for atomic locked RMW on the owning side, single atomic Phase-3 DB-ownership handoff milestone, one-writer contract documented and enforced in code
**Residual:** The Phase-3 handoff is itself the single riskiest cutover for the RMW family — a botched handoff (e.g. a lingering Python writer, or WAL/journal-mode mismatch between drivers) could momentarily create two writers. Mitigate by doing it as one atomic milestone with the old writer provably stopped first, and validate WAL/pragma parity between Python sqlite3 and modernc.org/sqlite. modernc is ~2x slower than CGO mattn — validate FTS5/vector latency under load, but correctness is unaffected.
**Confidence:** High — one-writer + wire-seam is the textbook fix and the strategy doc already codifies it as non-negotiable; residual is concentrated in executing the single handoff cleanly.

### (7) Provider-SDK / MCP spec churn — MCP 2026-07-28 deprecated roots/sampling/logging on a ~12-month window; sampling-with-tools was flagged unimplemented; SDK APIs and semconv evolve.
**Solution:** Wrap all external SDKs behind thin internal interfaces so spec moves are localized to one adapter each. For LLM: one provider-neutral interface (e.g. Stream(ctx, req) yielding TextDelta/Reasoning/ToolCall/StepFinish/Finish/Usage events) with an anthropic-sdk-go impl and an openai-go/v3 impl (Chat Completions for OpenRouter/Ollama/compat via option.WithBaseURL; Responses API native-OpenAI only). For MCP: consume the official modelcontextprotocol/go-sdk (GA v1.x, hard back-compat guarantee) behind a TAG-owned client/server facade, pin a protocol version, and budget periodic version bumps. For observability: adopt go.opentelemetry.io/otel (GA) but port the gen_ai.* attribute table + a pinned SEMCONV_VERSION as plain constants (no Go module exports GenAI semconv). Add cenkalti/backoff only for orchestration-level retry the per-request SDK retry doesn't cover.

**Why it works:** A thin adapter is the standard containment for churn: the ~97 commands depend on TAG's stable internal interface, so a deprecation or SDK API change touches one file, not the whole surface. The premise is also partly stale in TAG's favor — the audit's cited MCP gaps are CLOSED: sampling-with-tools (CreateMessageWithTools) landed in go-sdk v1.4.0, and client-side OAuth (Auth-Code+PKCE, Enterprise Managed Auth) stabilized in v1.5.0 with Client-Credentials in v1.6.0, directly enabling PRD-074. The official SDK's explicit backward-compat guarantee plus MCPGODEBUG escape hatches give a managed deprecation runway rather than a cliff. Both LLM SDKs are GA, permissively licensed, and provide per-request retry/backoff for free.
**Tools:** anthropic-sdk-go (MIT, GA v1.55.x, Go 1.24+), openai-go/v3 (Apache-2.0, GA), modelcontextprotocol/go-sdk (Apache-2.0, GA v1.x, back-compat guarantee; OAuth+sampling-with-tools now shipped), go.opentelemetry.io/otel + OTLP exporters (Apache-2.0, GA) with hardcoded gen_ai.* constants + pinned SEMCONV_VERSION, cenkalti/backoff/v4 (MIT — orchestration-level retry only), thin in-house provider/MCP adapter interfaces (crush/fantasy as reference design, not a dependency)
**Residual:** MCP protocol 2026-07-28 formally deprecates roots/sampling/logging and MCPGODEBUG back-compat hatches are slated for removal by ~v1.9.0 — requires periodic, budgeted version bumps. Go version floors differ (anthropic-sdk-go 1.24+, openai-go 1.22+) — target 1.24+. GenAI semconv has no published Go constants module (Development status) — the hardcoded table is a maintenance item (but pre-existing in Python). Responses API is not portable — must route native-OpenAI to Responses and everything else to Chat Completions.
**Confidence:** High — thin-adapter containment is proven, both LLM SDKs and the MCP SDK are GA with the MCP gaps already closed, so churn is a bounded, budgetable maintenance cost rather than a structural risk.

## Adopt from OpenCode

- Client-server split with a typed API as the ONLY contract: run the agent as a headless local server (`tag serve`) and make TUI/editor/SDK/CI all thin clients — define the API once and generate the OpenAPI spec + client SDK from it (opencode uses Effect httpapi→OpenApi.fromApi; in Go use huma/oapi-codegen). crush validates this with a Unix-socket/named-pipe daemon whose clients share workspace state keyed by cwd.
- Provider-neutral LLM core as DATA + per-provider PROTOCOL adapters (packages/llm design): one portable serializable Request/Message/Event/Tool schema, with all provider quirks isolated in inspectable protocol adapters (anthropic-messages, openai-chat, openai-responses, gemini, bedrock-converse, openai-compatible). Copy this schema-first boundary verbatim since Go has no Vercel AI SDK and must own this layer anyway.
- Prompt-caching-by-default with a cache breakpoint at the last-user-message boundary so every intra-turn tool round-trip reuses one cached prefix (Anthropic 1.25x write / 0.1x read means a single reuse already wins).
- Agent loop as an explicit continue|compact|stop state machine with overflow-driven compaction and doom-loop detection (DOOM_LOOP_THRESHOLD = 3 identical repeated tool calls). Small and clean to port to Go.
- The CONTEXT.md 'System Context / Context Epoch' model: typed Context Sources (value + stable key + codec + baseline/update renderers), immutable Context Epochs aligned to the provider prompt-cache boundary, and mid-conversation system messages emitted when observed state changes — the most sophisticated durable-context design in the audit; maps directly onto Go structs/interfaces.
- One unified tool interface implemented identically by built-in AND MCP tools (crush: Info()/Run(ctx,ToolCall)/ProviderOptions(); MCP tools wrapped as mcp_<server>_<tool>), with every Run() gated through a permission service (safe-tool whitelist) and per-call state threaded via typed context keys.
- Rule-based wildcard permission engine (match permission+pattern → ask/allow/deny with async approval), plus build/plan agent modes, external_directory gating, and a .env read guard — security-forward and trivially portable.
- Durable state in embedded SQLite with WAL + migrations, persisting the full session so it can be reconstructed/reconnected; Thread/Turn/Item as the persistence schema (Codex pattern, echoed by opencode).
- Markdown-with-frontmatter custom commands with argument templating surfaced in a command palette, and Claude-compatible Agent Skills (SKILL.md discovered from .claude/skills, .agents/skills, .opencode/skill) for cheap interop wins.
- MCP via the official protocol as BOTH client and server (stdio + SSE + streamable-HTTP, tool-name namespacing, OAuth-for-remote with a local callback listener) and ACP for editor integration — table stakes, not roadmap.
- Distribution matrix as a first-class deliverable: prebuilt per-platform binaries via curl|bash + GitHub releases AND an npm optionalDependencies shim, plus brew/scoop/winget/nix/AUR — Go gets this nearly for free via GoReleaser and go:embed for assets (cleaner than Bun's file-import trick).

## Learn from Hermes (design only — port NONE of its Python)

- Port the declarative ProviderProfile + api_mode dispatch (providers/base.py) almost verbatim to Go: one dataclass declares auth/endpoints/quirks/api_mode instead of 20+ boolean flags; back it with anthropic-sdk-go (anthropic_messages) + openai-go (chat_completions and Responses/codex) + base_url override for OpenRouter/Ollama. This IS the 'thin internal provider interface' the strategy doc mandates.
- Reuse the synchronous bounded agent-loop shape (conversation_loop.py): while(iters<max && budget.remaining || grace){ call; dispatch tools; append tool_result } with an IterationBudget (token/step budget distinct from iteration count), a one-turn 'grace call' so the model can wrap up after budget exhaustion, and a cooperative interrupt flag — confirms the ~200-LOC hand-rolled Go loop is faithful, not a simplification.
- Adopt evidence-based verification / completion contracts (0.18: verification_evidence.py, /goal contracts, pre_verify hook): the agent proves 'done' by running the project's real checks and recording a verification-evidence ledger rather than asserting completion — a genuinely novel, portable differentiator that elevates TAG's coding-agent story.
- Treat MCP as a two-way citizen: same process is an MCP client (registers remote tools as if built-in) AND an MCP server (exposes its own conversations/tools) — target both with modelcontextprotocol/go-sdk.
- Keep the central self-registering tool registry + toolset grouping (tools/registry.py, toolsets.py), but in Go use explicit register() calls (not AST scanning) keyed by toolset with an availability predicate per tool.
- Keep the UI<->runtime seam as newline-delimited JSON-RPC over stdio (Hermes Node/Ink <-> Python tui_gateway) — real-world proof of the strangler wire-protocol seam; TAG's Go core should expose the same so a single bubbletea front-end and external clients drive it.
- Keep the SQLite + FTS5 SessionDB state model and port to modernc.org/sqlite (CGO_ENABLED=0), preserving the FTS5 session-search schema; add sqlite-vec only if scale demands.
- Keep supply-chain discipline: Sigstore/cosign release signing (Hermes attaches .sigstore.json per release), exact-pinned deps, and lazy-optional providers — but translate to goreleaser + cosign keyless + SLSA and STATICALLY link all providers (runtime lazy-pip-install is incompatible with a static binary).
- AVOID porting: the ~19.7k-LOC gateway/run.py god-file and ~20 messaging-platform adapters (TAG is not a chat-ops bridge), the four-frontend UI sprawl (prompt_toolkit CLI + Ink TUI + Electron + PTY dashboard — ship ONE Go TUI), runtime lazy-pip-install, and feature bloat (pets/achievements/spotify/homeassistant/browser stacks).
- Consider the aux-model-routed background self-improvement fork (background_review.py) and the atomic batch memory tool (operations[] applied atomically against a char budget) — both cost/reliability wins that port directly.

## Open decisions the migration plan must resolve

- Adopt charm.land/fantasy directly for speed, or hand-roll the provider interface over the official SDKs? Fantasy is PREVIEW-status, wraps Charm FORKS of the SDKs, and is FSL-1.1-MIT (not OSI-approved at release) — the recommendation leans hand-rolled-on-official-SDKs for a shipped binary, but this trades weeks of build effort and must be decided before Phase 1.
- Vector storage backend: (A) modernc.org/sqlite + embeddings in a BLOB column + in-Go cosine (simplest, one file, matches shipped behavior) vs (B) ncruces/go-sqlite3 + sqlite-vec (real ANN, keeps one file, but a pre-1.0 extension on a WASM/wazero driver DIFFERENT from modernc) vs (C) chromem-go (splits state into a second RAM store). Recommendation is (A) now, (B) past ~100k vectors — but the choice sets the SQLite driver for the WHOLE project and must be made early.
- Local offline embeddings: accept the loss and default to a provider embedding API (network + per-embed cost + offline failure mode), or preserve offline capability via nlpodyssey/cybertron (pure-Go MiniLM, slow/low-maintenance) behind a build tag, or a CGO ONNX build variant? This is the single genuine capability regression and needs an explicit product decision + a dimension-migration plan (384→1536 rebuild + model/dim tag).
- Server API contract shape: spec-first (huma/oapi-codegen generating handlers+client+OpenAPI, opencode-style) vs code-first (chi handlers + swaggo annotations, crush-style)? This decides how TUI/editor/CI/remote clients are generated and is foundational to the client-server split.
- In-process DAG queue: build bespoke on goroutines+channels+SQLite, or adopt Azure/go-workflow for topology (small maintainer base)? No off-the-shelf pure-Go embedded durable DAG queue exists — scope this as real engineering and decide the persistence/replay schema (ties into PRD-032/113 time-travel).
- Plugin/extensibility model: TAG's existing hooks/plugins and opencode's runtime-JS plugins do NOT map to a static Go binary. Decide between out-of-process plugins (hashicorp/go-plugin/gRPC subprocess), a scripting VM (risor/starlark/lua/wasm), or shell/HTTP hooks only (Claude Code style) — this is the one feature that fundamentally resists the single-binary goal.
- Exact Bubble Tea v2 / lipgloss v2 / bubbles v2 pinned versions and API alignment — sources conflict on the latest tag; confirm on the releases page and run the week-one streaming spike BEFORE committing Phase 4 TUI work.
- Which distribution channels are Phase-1 must-haves (curl|bash + Homebrew) vs later discoverability (scoop/winget/nix/AUR/npm shim), and the exact kill-date milestone + production-soak criteria that gate retiring the transitional Python/Hermes sidecar (governance gate for the moral-hazard risk).
- MCP protocol version to pin now (2025-11-25 vs the 2026-07-28 that deprecates roots/sampling/logging) and the cadence for budgeted version bumps before MCPGODEBUG back-compat hatches are removed (~v1.9.0).
- Scope cut line: which of the ~72 features are in-scope for native parity vs dropped/deferred (e.g. Hermes-style messaging-gateway analogs, batch trajectory/training surfaces, desktop app PRD-007) — the golden-transcript fixture set must be declared exhaustive against this line, since it is the migration completeness ledger.
