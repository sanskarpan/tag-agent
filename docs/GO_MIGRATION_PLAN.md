# TAG — Complete Python→Go Migration Plan

> The full feature-by-feature migration plan for all ~72 implemented features (103 commands) from
> the Python control-plane + bundled Python Hermes runtime to a **native single-static-binary Go**
> harness that owns its runtime. Grounded in `GO_MIGRATION_RESEARCH.md`. Strangler-fig, never big-bang.

> **Status (2026-07-08): implemented.** The native Go harness landed as [`tag-go/`](../tag-go/)
> (87 top-level commands, 28 packages, both tracks done). Current per-subsystem status lives in
> [`tag-go/MIGRATION_STATUS.md`](../tag-go/MIGRATION_STATUS.md); benchmarks and behavioral parity
> vs the Python edition in [`COMPARISON_REPORT.md`](../COMPARISON_REPORT.md). The as-built layout
> diverges from the plan below in one major way: the port went straight to a native runtime — the
> transitional Python↔Go wire seam (`internal/runtime/`, `hack/hermes-runtimed/`) was never
> needed. The managed-Hermes passthrough commands and desktop packaging were deliberately not
> ported; live-model execution defaults to an offline `echo` provider.

## Approach

Strangler-fig migration to a native single-static-binary Go harness that OWNS its runtime, sequenced so the distribution/UX win ships at the END of Phase 1 (~2 months) — before the runtime is fully ported — and the product never feature-freezes. Two tracks run in parallel: TRACK A (mechanical stdlib->Go control-plane ports of the ~72 features / 103 commands: cobra, modernc.org/sqlite, net/http+chi, yaml.v3, crypto, os/exec, goroutines) and TRACK B (genuinely NEW runtime ownership that TAG delegates to Hermes today: multi-provider LLM + streaming + tool-exec + agent loop + MCP client/server + permission gate). Two hard invariants govern every phase: (1) exactly ONE writer/owner of the SQLite state at all times — the Python<->Go seam is a wire protocol (JSON-RPC/HTTP over loopback|stdio), NEVER shared DB-file access, protecting the read-modify-write race hardening earned across three bug bashes; (2) all Hermes coupling flows through the three chokepoints in src/tag/core/profile.py (run_hermes L199, run_profile_hermes L210, run_profile_python L228), collapsed behind ONE versioned RuntimeService interface so replacement is localized, measurable, and the same typed schema backs both the transitional wire seam and the public huma HTTP API. The Go binary starts as a thin front-door wrapping today's Python+Hermes over that wire, then absorbs subsystems inward (LLM -> agent loop -> MCP/tools -> SQLite state -> servers/observability -> TUI), porting command-by-command behind a golden-transcript harness until Hermes and finally the Python control plane are deleted. Language is Go (decided): 95% of Rust's distribution win at materially higher velocity for a bus-factor-of-one maintainer, plus official GA anthropic-sdk-go + openai-go/v3 + Google-co-maintained MCP go-sdk that de-risk runtime ownership to a PORT, not research. Parity is proven by golden transcripts PLUS isolated-TAG_HOME sandbox runs (the v0.8.1 audit found ~30 dispatch-layer bugs masked by green unit tests) — never by the unit suite alone.

## Resolved architecture decisions

### (1) Adopt charm.land/fantasy directly, or hand-roll the provider interface over the official GA SDKs?
**Decision:** Hand-roll one TAG-owned provider-neutral interface — Stream(ctx, Request) -> <-chan Event (TextDelta/Reasoning/ToolCall/StepFinish/Finish/Usage) — with two adapters: internal/llm/anthropic over anthropics/anthropic-sdk-go v1.55.x (Messages.NewStreaming + Message.Accumulate) and internal/llm/openai over openai/openai-go/v3 v3.41.x (Chat Completions + option.WithBaseURL for OpenRouter/Ollama/compatible; Responses API only for native OpenAI). Port Hermes's declarative ProviderProfile (providers/base.py: auth/endpoints/quirks/api_mode as one struct) verbatim into internal/llm/profile.go. Use fantasy/crush as DESIGN reference only — never a dependency. invopop/jsonschema for tool schemas; cenkalti/backoff/v4 for orchestration-level retry only.

**Rationale:** fantasy is PREVIEW-status, wraps Charm FORKS of the two SDKs, and ships under FSL-1.1-MIT (not OSI-approved at release) — unacceptable for a shippable static-binary product and a supply-chain/licensing liability. The official SDKs are GA and permissive (anthropic-sdk-go MIT, openai-go/v3 Apache-2.0) and provide per-request retry/streaming/tool-use for free, reducing runtime ownership to a PORT not RESEARCH. A thin in-house interface is also the churn-containment boundary (risk 7): a spec/API change touches one adapter, not the ~103 commands. Grounded: today all LLM work is delegated out via profile.py run_hermes/run_profile_hermes/run_profile_python (L199/L210/L228) — the interface simply replaces those three chokepoints.
**Blocks:** Phase 2 (own the LLM layer + agent loop). Must be locked before Phase 2 starts; interface stub can front hermes-runtimed in Phase 1.

### (2) Vector storage backend: (A) modernc.org/sqlite + embeddings in a column + in-Go cosine, (B) ncruces/go-sqlite3 + sqlite-vec, or (C) chromem-go?
**Decision:** Option A now, and this decision also LOCKS modernc.org/sqlite as the single SQLite driver for the entire project (CGO_ENABLED=0, FTS5/JSON1/RTree compiled in). Store embeddings in a BLOB column of float32 (migrating semantic_memories.embedding_json from TEXT/JSON to BLOB, keeping a JSON read-compat path) and do brute-force cosine in Go behind a VectorIndex interface. Defer Option B (sqlite-vec on the ncruces/wazero driver) behind that interface as a documented scale trigger past ~100k vectors per profile. Reject Option C (chromem-go) as the primary store because it splits state into a second RAM/on-disk store, violating the one-file / single-writer mandate.

**Rationale:** Shipped behavior is already brute-force cosine (semantic_memory.py _cosine_sim L1040) over a JSON column at hundreds-to-low-thousands of vectors per profile — no ANN engine is needed, so A is a 1:1 port with zero new infra. modernc compiles FTS5 in by default so the whole PRD-025/066/072 keyword+hybrid path (BM25 L273 + RRF k=60 L425) is CGO-free. Critically, the driver choice is project-wide (it is the same store that holds runs/steps/spans/queue), so choosing A keeps ONE driver, ONE file, ONE writer; B would force a second, WASM-based driver (wazero) that conflicts with modernc's semantics — only justified if scale ever demands real ANN. tool_retrieval.py's ChromaDB usage (≤500 tools) collapses into the same in-Go cosine, deleting the Chroma dependency entirely.
**Blocks:** Driver choice blocks Phase 1 (cross-compile build config). Vector-column/cosine port blocks Phase 3 (SQLite state + memory ownership).

### (3) Local offline embeddings: accept the loss (provider API), preserve offline via pure-Go MiniLM behind a build tag, or a CGO ONNX variant? Plus the dimension-migration plan.
**Decision:** Default to a provider embedding API behind an Embedder interface (internal/memory/embed) — openai-go embeddings as the default impl (Voyage/Cohere/Gemini pluggable). Keep the EXISTING FTS5 graceful-degradation as the primary offline path (embed_text() already returns None -> search_memories() FTS5 fallback, semantic_memory.py L1073). Offer pure-Go offline MiniLM (nlpodyssey/cybertron, or knights-analytics/hugot GoMLX) behind a build tag for air-gapped users who accept the perf hit. Do NOT ship a CGO ONNX variant (re-breaks static cross-compile). Dimension migration: add a vector_store_meta row carrying {model, dim}; on model/dim change run a `mem rebuild-embeddings` migration; enforce a mismatched-dim guard on read (384-dim MiniLM vs 1536/3072-dim provider) that forces rebuild-or-degrade rather than silently comparing incompatible vectors.

**Rationale:** This is the single genuine capability regression, but it is bounded: the Python code ALREADY degrades to FTS5 when sentence-transformers is absent, so that same fallback simply becomes the primary offline path — no user-visible regression for the common case. Provider embeddings add a network dependency + per-embed cost + an offline failure mode, which is why FTS5-degradation and the optional build-tagged local model both remain. The build tag keeps the default binary CGO-free and small while still offering a genuine offline story. The dim guard prevents a silent-corruption class when the model changes.
**Blocks:** Phase 3 (memory subsystem port). The dimension-migration + meta schema must land with the semantic-memory cutover.

### (4) Server API contract: spec-first (huma/oapi-codegen) vs code-first (chi + swaggo)?
**Decision:** Spec-first via danielgtaylor/huma v2 mounted on a go-chi/chi v5 router. Define the API once as Go operation structs; huma emits the OpenAPI 3.1 spec + request/response validation, and the spec drives generated TUI/editor/CI/remote client SDKs. Reject code-first swaggo (annotations drift from implementation) and reject standalone oapi-codegen (heavier codegen ceremony than huma's define-in-Go-emit-spec model). The same typed schema is shared with the RuntimeService wire seam so the transitional protocol and the public HTTP API do not diverge.

**Rationale:** Adopt opencode's 'typed server is the only contract' boundary: TUI, editor, CI, and remote clients are all thin clients generated from one spec. huma is the Go-native sweet spot between the two poles — you write Go (like code-first) but get a guaranteed-accurate emitted OpenAPI + validation + client generation (like spec-first), with no annotation drift and full chi compatibility (the router chosen for route-group middleware: auth/OAuth/HITL/rate-limit). This makes the client-server split (tag serve) a first-class product surface rather than an afterthought.
**Blocks:** Phase 4 (servers: webhook/gateway/dashboard + SSE). Contract shape should be sketched in Phase 1 so the front-door RuntimeService reuses the same types.

### (5) In-process DAG queue: build bespoke, or adopt a library (River/asynq/Azure go-workflow)?
**Decision:** Build a bespoke in-proc scheduler in internal/queue: a goroutine worker pool (errgroup, bounded concurrency) over channels, persisted to modernc.org/sqlite using the already-designed queue_jobs (with deps_json, db.py L119/L576) and queue_dags (L424) tables. Design: job rows with status pending/ready/running/done/failed; a scheduler goroutine promotes pending->ready when all deps_json predecessors are done; Kahn's-algorithm cycle detection at enqueue (matches PRD-033 cycle detection); event-sourced state transitions persisted for replay/time-travel (PRD-032/113). Reject River (Postgres) and asynq (Redis) — both break the single-binary mandate — and Azure/go-workflow (tiny maintainer base, in-memory-only topology).

**Rationale:** The research is explicit that no pure-Go embedded durable DAG-capable queue exists in 2026, so this is genuine engineering, not a library drop-in. The good news: the SQLite schema and semantics already exist in the Python impl (queue_jobs.deps_json + queue_dags + idx_qj_status L135), so the Go build ports proven data structures. Persisting transitions to SQLite (the same single-writer store) gives durability + replay for the time-travel PRDs without a second datastore. gocron handles cron (PRD-022) separately; this scheduler handles the dependency DAG (PRD-008/033).
**Blocks:** Phase 3 (own SQLite state + queue/DAG + cron). The persistence/replay schema must be settled here to support PRD-032/113.

### (6) Plugin/extensibility model for a static binary: go-plugin/gRPC, scripting VM, or shell/HTTP hooks only?
**Decision:** Shell/HTTP hooks only for v1 (Claude Code style), which also matches the SHIPPED model: a JSON plugin registry (cmd/routing.py _load_plugin_registry L640) plus lifecycle hooks that fire shell commands / HTTP webhooks (cmd/workflow_mgmt.py _execute_hook L478, _fire_hooks L508). The real extensibility surface is MCP servers — out-of-process tools over stdio/HTTP/SSE via the official go-sdk (both directions). Add markdown-with-frontmatter custom slash-commands + Claude-compatible Agent Skills discovery (.claude/skills, .agents/skills) for no-code extension. Reject hashicorp/go-plugin/gRPC (per-plugin binaries defeat the single-file distribution win and complicate cross-compile) and an embedded scripting VM (risor/starlark/lua/wasm — new attack surface + maintenance) for v1; keep a wasm/starlark VM as a documented future option only.

**Rationale:** This is the one feature that fundamentally resists a static binary, so the decision follows the least-surprise path: preserve exactly what ships today (JSON registry + shell/HTTP hooks), which ports mechanically, and lean on MCP — which TAG must own as client+server anyway — as the genuine plugin bus. go-plugin would reintroduce multi-artifact distribution (the exact pain the migration exists to kill); a scripting VM is a large new security/maintenance surface unjustified for a bus-factor-of-one team at v1.
**Blocks:** Phase 4 (remaining commands: plugin/hooks/skills). MCP-as-extensibility is enabled earlier in Phase 3.

### (7) Exact Charm v2 pins + the week-1 TUI streaming spike.
**Decision:** Pin charmbracelet/bubbletea v2.0.0, lipgloss v2, bubbles v2, and glamour (latest GA), but treat the exact tags as confirm-at-build (v1->v2 has breaking Cmd/Msg/renderer/program-option changes). Gate all Phase 4 TUI work on a Phase-0 week-1 spike: a multi-pane live-token-streaming prototype (Msg-per-delta into a bubbles viewport, incremental glamour markdown) run at full throughput under SSH latency — an explicit GO/NO-GO. Document the ratatui/OpenTUI-Zig-behind-the-Go-front-end fallback but do NOT build it speculatively.

**Rationale:** The 'opencode abandoned Go TUI' story is a stale pre-v2 anecdote; the v2 Cursed Renderer retired that ceiling and charmbracelet/crush proves it in production for a same-class streaming agent. The residual risk is purely pin/API churn and TAG's specific multi-pane throughput, both cheaply retired by the week-one spike before any Phase 4 commitment. Worst case (spike fails) costs only a renderer swap behind the same Go front-end, kept low-probability by crush's production use.
**Blocks:** Phase 4 (full TUI). The GO/NO-GO spike itself runs in Phase 0 alongside the anthropic-sdk-go + MCP agent-loop PoC.

### (8) Phase-1 must-have distribution channels + the kill-date gate for the transitional Python/Hermes sidecar.
**Decision:** Phase-1 must-haves: curl|sh install.sh + a Homebrew tap + the thin npm/pip download-shim (the @openai/codex pattern — downloads the native binary), all via GoReleaser (CGO_ENABLED=0, -trimpath, ldflags -s -w) with checksums + syft SBOM + keyless cosign v3 + SLSA L3 + macOS notarization. Defer scoop/winget/nix/AUR/Docker to Phase 4-5 discoverability. Kill-date gate for retiring the transitional relocatable-CPython + Hermes sidecar (Phase 5): retire ONLY when all three hold — (a) 100% of the ~103 commands pass golden transcripts AND isolated-TAG_HOME sandbox runs (not the unit suite), (b) a production soak of >=4 weeks / >=2 minor releases with zero parity-regression bugs, and (c) an explicit funded native-parity sign-off. Green CI alone never triggers deletion.

**Rationale:** Phase 1 must kill Node>=20, system-Python 3.11-3.13 detection, and the network-dependent first-run pip build — curl|sh + brew + the download-shim achieve that immediately (widest realized-adoption channels per crush; the shim preserves the familiar entry point). The other channels are pure discoverability and can wait. The three-part kill gate directly answers the moral-hazard risk (risk 5): tying retirement to a funded milestone + production soak converts 'urgency' from willpower into a governance gate, so the transitional sidecar cannot quietly ossify.
**Blocks:** Phase 1 (channels ship here) and Phase 5 (the gate authorizes deleting Hermes + the Python control plane).

### (9) Which MCP protocol version to pin now, and the bump cadence?
**Decision:** Pin protocol version 2025-11-25 now, consumed via modelcontextprotocol/go-sdk v1.6.1 (GA, back-compat guarantee; sampling-with-tools since v1.4, client OAuth PKCE since v1.5, client-credentials v1.6). Wrap it behind a TAG-owned client+server facade (internal/mcp) so the version is a single constant. Schedule a budgeted bump to 2026-07-28 (which deprecates roots/sampling/logging on a ~12-month window) BEFORE the MCPGODEBUG back-compat hatches are removed (~go-sdk v1.9.0).

**Rationale:** 2025-11-25 is the stable version that still supports roots/sampling/logging, giving the widest current-server compatibility while the migration is in flight; jumping straight to 2026-07-28 would take deprecations early for no benefit. The official SDK's hard back-compat guarantee plus MCPGODEBUG hatches provide a managed runway rather than a cliff, and the thin facade (risk 7 containment) means the eventual bump touches one adapter. This enables PRD-014 (client/registry) and PRD-006 (server/gateway) in Phase 3.
**Blocks:** Phase 3 (own MCP client+server). The scheduled bump to 2026-07-28 is a budgeted maintenance item before go-sdk v1.9.0.

## Master phased plan

### Phase 0 — Decouple Hermes behind a wire protocol (the convergent first cut)  (3-5 weeks)
**Goal:** Collapse ALL Hermes coupling behind one versioned RuntimeService wire interface so Python stops inlining `python -c` strings into the Hermes venv, establish the single-SQLite-writer contract, and build the golden-transcript capture harness. Simultaneously the last Python-side refactor AND step one of owning the runtime — throwaway-free. Lock the resolved decisions that block later phases.

**Features / domains:**
- No feature ports natively yet — pure internal refactor; user-visible behavior is byte-identical (patch release).
- RuntimeService seam at profile.py L199/L210/L228 (run_hermes / run_profile_hermes / run_profile_python) — collapse to one interface: run turn / stream / dispatch tool / auth ops.
- run_profile_python's inline `python -c` into the Hermes venv (load_model_inventory L483 importing hermes_cli.inventory) becomes a typed RuntimeService.ModelInventory RPC on the sidecar — the single hardest seam mapping.
- Stand up hack/hermes-runtimed thin Python adapter exposing Hermes over the wire; control plane becomes its sole client.
- Document + enforce the one-writer SQLite contract (runtime returns data over the wire, never opens tag.sqlite3).
- Golden-transcript capture harness across all 103 commands (args/env/stdin -> stdout/exit/DB-delta) in isolated TAG_HOME sandboxes.
- LOCK decisions: #1 provider interface shape, #2 modernc.org/sqlite as the project-wide driver (blocks P1 build config), #7 the week-1 TUI GO/NO-GO spike, #9 MCP protocol pin 2025-11-25.
- Week-1 de-risking spikes: (a) bubbletea v2 multi-pane live-token-streaming prototype under SSH latency (explicit GO/NO-GO); (b) ~50-LOC anthropic-sdk-go + MCP go-sdk agent-loop PoC.

**Deliverable:** A released patch where 100% of Hermes invocations flow through RuntimeService, zero inline-Python-string exec remains, and golden transcripts are captured for all 103 commands and pass against the current Python impl. Two spike verdicts documented.
**Keeps shipping:** Yes — internal refactor released as a normal patch version; no install-story change.
**Risks:** Hidden stateful coupling in inline-Python strings assuming shared process memory with Hermes.; Streaming (token-by-token) must survive the wire boundary without added latency.; The ModelInventory RPC is the one place a 'just shell out' shortcut fails — must be a typed method.
**Exit criteria:** All Hermes calls via RuntimeService; zero inline `python -c` into the venv; golden transcripts captured for 100% of commands and passing; existing ~678-test suite green; TUI streaming spike returns GO (or the ratatui fallback is chosen); anthropic/MCP PoC confirms the provider+agent-loop shape.

### Phase 2 — Own the LLM layer + agent loop natively in Go  (6-9 weeks)
**Goal:** Move the core agent runtime (multi-provider LLM calls, streaming, tool dispatch, inner+outer agent loops) from Hermes into native Go so the binary stops needing Hermes for the primary execution path. The single largest NEW-BUILD in the migration (TAG owns ZERO of this today).

**Features / domains:**
- NEW multi-provider LLM + streaming + tool-use replacing Hermes `chat -q PROMPT -Q`: internal/llm provider iface + anthropic adapter (Accumulate, prompt-cache-by-default breakpoints — PRD-030) + openai adapter (Responses-vs-ChatCompletions api_mode branch, OpenRouter/Ollama base-URL override).
- Declarative ProviderProfile port (Hermes providers/base.py) + parse/format_model_ref grammar; model catalog reconstruction from ProviderProfile + provider list-models + go:embed pricing table (the load_model_inventory data-source gap).
- Hand-rolled inner agent loop: continue|compact|stop + NEW doom-loop(3) + IterationBudget + one-turn grace + cooperative interrupt + overflow compaction (context-epoch model).
- Unified builtin+MCP tool interface + rule-based permission gate (NEW; design ref crush/opencode) — wraps every tool.Run().
- Task routing engine native: route / assignments / models / set-model / submit (ThreadPoolExecutor -> errgroup, commit-per-step preserved) / openrouter-models / runs; PRD-031 route-fallback (cycle detection).
- PRD-017 benchmark/compare (in-memory ProviderProfile clone instead of temp-profile HOMEs) + normalize_chat_output/infrastructure_failure_reason status-classification compat shim.

## Golden-transcript parity harness

Parity is proven by a golden-transcript harness, never by trust or the unit suite (the v0.8.1 audit found ~30 dispatch-layer bugs masked by green pytest). Structure: testdata/golden/<domain>/<command>/<case>/ where each fixture is a self-contained sandbox capturing (argv + env + stdin + seeded TAG_HOME/DB/repo state) -> (stdout, stderr, exit code, DB-delta over the touched tables, and file/config-delta). Fixtures are captured from the RUNNING Python impl in an isolated TAG_HOME in Phase 0 for all 103 commands; the Go impl must replay them byte-for-contract before the corresponding Python code is deleted (Phase 5 kill-gate). Both --json and plain-text modes are captured, with NO_COLOR=1 and non-TTY so colored/Rich paths degrade deterministically.

REQUIRED harness controls for nondeterminism: (1) FROZEN CLOCK injected via a clock interface/env so confidence-decay, tier classification, temporal valid-at, cron matching, and all created_at/accessed_at timestamps are reproducible. (2) SEEDED/NORMALIZED IDs — uuid4().hex[:16]/[:12] and full-uuid IDs are either RNG-seeded or normalized in the diff. (3) EXACT timestamp format assertion '2006-01-02T15:04:05.000000+00:00' (microseconds + '+00:00', not RFC3339 Z) because list_facts_at compares lexically. (4) BANKER'S ROUNDING (round-half-even to 4 dp) helper — naive math.Round fails. (5) PROVIDER cassettes via go-vcr keyed by profile so LLM/streaming suites are offline-deterministic. (6) PATH-shimmed stub gh/git/hermes/npm binaries recording argv.

Fixture classes: (A) PURE/DETERMINISTIC — routing, memory arithmetic, SARIF/CI-log parsing, cron matching, scaffold/pipeline YAML, pricing globs, security scans, config/.env writes — capturable and byte-exact immediately. (B) STUBBED-RUNTIME — commands whose only nondeterminism is the LLM: run against a canned RuntimeService/recorded cassette (eval-judge, memory-extract, issue-solve, swe-solve, swarm coordinator). (C) NEW BEHAVIORAL fixtures with no Python baseline (authored fresh in Phase 2): streaming event ORDERING, multi-turn tool round-trips, doom-loop trip at 3 identical calls, IterationBudget exhaustion + grace, prompt-cache breakpoint reuse, Responses-vs-ChatCompletions routing, MCP handshake + tool-call round-trip, os/exec pgid-kill leaves no orphans, permission ask/allow/deny matrix, HMAC vectors, SSRF rejection (loopback/169.254.169.254/private/DNS-rebind/public-302->loopback). DELIBERATE re-baselines (content changes, not byte-parity): eval-ci/agentic-ci GitHub Action scaffolds switch from `pip install tag-agent` to native install channels; vector ranking (Chroma HNSW -> exact cosine) captured in keyword-fallback mode. Every command additionally gets an isolated-TAG_HOME sandbox smoke run before its Python counterpart is deleted. Fixtures for the ~5 prd_clusters<->ci.py/swe_harness signature-mismatch bug clusters are captured only AFTER reconciling the correct contract by RUNNING each command.

## The single atomic SQLite ownership handoff

Invariant across the whole migration: exactly ONE writer/owner of tag.sqlite3 at all times; the Python<->Go seam is the wire protocol, NEVER shared DB-file access. Phases 0-2: the Python control plane is the sole writer. The RuntimeService seam returns data over the wire and NEVER opens the DB; native llm/agent/tool packages must NOT open tag.sqlite3 before the handoff. All runs/steps/loop_*/swarm_*/split_* writes stay on Python until Phase 3.

The handoff is ONE atomic Phase-3 milestone, not a gradual split: (1) Port core/db.py schema to internal/store on modernc.org/sqlite (WAL, FTS5, JSON1, all idempotent CREATE-IF-NOT-EXISTS + ALTER-if-missing migrations replayed verbatim, incl. the PRD-021/032/033/044 blocks, deps_json, queue_dags, spans/trace_snapshots/cost columns). (2) Validate WAL/journal_mode + busy_timeout + pragma parity modernc-vs-Python-sqlite3, and preserve open_db's 20x0.1s locked-retry loop and the once-per-24h _prune_old_spans maintenance gate (so reads never take a write lock). (3) At cutover, stop the Python writer PROVABLY (control plane goes read-only / is shut down), then Go opens the store as sole writer — the wire protocol remains the only cross-language seam; there is never a window where both languages hold the DB. (4) Route everything through the single internal/store handle with flock + os.Rename atomic RMW (fixes the mem2/graph raw-connect inconsistency and the B005 lost-update class in one move; a strict upgrade over Python's fcntl which no-ops on Windows).

Two nuances: (a) memory read-paths that bump accessed_at/access_count are WRITES-on-read and must be honored by the single writer; (b) kanban.db is a SEPARATE per-board interop store deliberately Hermes-schema-compatible — during transition it legitimately has two writers (TAG + Hermes gateway), so the single-writer mandate applies ONLY to tag.sqlite3; kanban.db keeps WAL + BEGIN IMMEDIATE until Hermes is deleted in Phase 5, at which point it too becomes single-writer under Go.

## Distribution-first plan

Ship the native Go single binary as the install artifact at the END of Phase 1 — BEFORE the runtime is ported — fronting the transitional Python/Hermes backend over the P0 wire. This front-loads the #1 adoption lever (Codex saw ~510x download growth post-native) ~2 months in, while the remaining ~6-10 months hollow out the runtime behind an already-shipping native install experience. Phase-1 must-have channels (decision 8): curl -fsSL .../install.sh | sh + a Homebrew tap + a thin npm/pip download-shim (the @openai/codex pattern — the shim's only job is to fetch the right prebuilt binary for GOOS/GOARCH and exec it, preserving the familiar entry point; it keeps the 128+signo signal-exit convention for CI/set -e parity). Build: GoReleaser + CGO_ENABLED=0 (enabled by the modernc.org/sqlite driver lock) + -trimpath + ldflags -s -w for trivial cross-compile of darwin(arm+intel)/linux(musl+glibc)/windows from one machine. ## Explicit non-goals

- Do NOT ship a standalone hardened-Python (PyApp/uv) launcher as the product end-state — it caps at ~40-60% of the distribution win and risks the aider bridge-becomes-destination trap. PyApp appears ONLY as an internal, time-boxed, kill-dated backend-sidecar inside the Go binary during P1-P4.
- Do NOT split the single SQLite database across Python and Go via shared file access — the seam is the wire protocol only; exactly one writer at all times, or the RMW race family the repo hardened across three bug bashes returns.
- Do NOT adopt eino/langchaingo or any agent framework as a core dependency — hand-roll the ~200-LOC agent loop (crush/goose precedent).
- Do NOT adopt charm.land/fantasy — PREVIEW status, wraps Charm forks of the SDKs, FSL-1.1 licensing; hand-roll the provider interface over official GA anthropic-sdk-go + openai-go/v3 (fantasy/crush as design reference only).
- Do NOT choose mattn/go-sqlite3 (CGO) — it re-breaks CGO_ENABLED=0 static cross-compile; use pure-Go modernc.org/sqlite even at ~2x query cost. Do NOT introduce a second WASM-based driver (ncruces/wazero) except behind the deferred sqlite-vec scale trigger.
- Do NOT use chromem-go as the primary vector store (splits state into a second RAM/on-disk store, violates one-file/single-writer) — brute-force cosine over a BLOB column in the one store.
- Do NOT attempt native OS-level syscall sandboxing in Go beyond the landlock+seccomp+nftables restricted tier + docker; gVisor/Firecracker are documented later tiers. If in-process syscall sandboxing of untrusted code becomes a first-class must-own requirement, that single trigger is the only reason to reconsider Rust.
- Do NOT use path/filepath.Match or path.Match for model-id/blocked-pattern globs — Python fnmatch '*' crosses '/', Go's does not; use gobwas/glob or a fnmatch-compat matcher (silent mis-resolution otherwise).
- Do NOT use go's OTLP protobuf exporter for the export commands — hand-roll the two distinct OTLP-JSON encoders for byte-exact stdout; use the otel SDK only for optional live in-loop span emission.
- Do NOT unify the two OTLP-JSON encoders, the two cost tables (_COST_TABLE per-1K vs PricingEntry per-1M), or the two GitHub-Action scaffolds (ci.py vs eval_ci.py) — each has distinct semantics and its own golden fixture.
- Do NOT 'fix' deliberate approximations during the port: BM25 corpus-size IDF, banker's rounding, the trace_snapshots single-row dedupe, workspace bytes/4 token estimate, cron fixed day-deltas (monthly=30) — parity requires reproducing them bug-for-bug.
- Do NOT simplify the security-critical guards: notify SSRF connect-time IP-pin + per-redirect-hop re-validation, webhook HMAC constant-time compare + 10MiB cap + 401-before-enqueue, SARIF repo-root write-containment, hooks shell-quote injection guard, findings/logs never carrying secret values, .env written 0600 BEFORE secrets touch disk.
- Do NOT decompose the annotation-queue atomic UPDATE...RETURNING claim into SELECT-then-UPDATE (reintroduces the lost-update race); do NOT write tag.yaml via unlocked yaml marshal (route through internal/config flock+os.Rename).
- Do NOT feature-freeze for a big-bang rewrite — port command-by-command behind golden transcripts, dual-maintaining until each reaches parity.
- Do NOT trust the unit suite as the parity gate — verify by RUNNING commands in isolated TAG_HOME sandboxes.
- Do NOT delete Hermes or the Python control plane until the 3-part kill-gate holds (100% golden+sandbox pass AND >=4-week / >=2-release production soak AND funded native-parity sign-off) — green CI alone never triggers deletion.
- Do NOT port the ~5 prd_clusters<->ci.py/swe_harness signature-mismatch bugs as-is — pick one canonical contract per function, verify by RUNNING, then freeze the golden.
- Do NOT port the 55 planned features (PRD-073..127) — they are out of migration scope.
